# Super-Admin Bootstrap Runbook

**When to use:** First-time go-live for a fresh production deployment, or when the entire app needs a clean reset.

**What it does:**

1. Truncates ALL operational tables in production (organizations, schools, users, students, sessions, all summaries, audit_log — see `scripts/production_wipe.py` for the full list).
2. Inserts the first CEO account atomically with the truncate, using the four `SUPER_ADMIN_*` env vars as credentials.
3. Leaves only static reference data behind: `skill_domains`, `skills`, `benchmarks`, `app_settings`, `role_permissions`.

**What it does NOT do:**

- Does not rotate Supabase database passwords (do that separately in the Supabase dashboard).
- Does not clear Render env vars after running (you must do this manually — see Step 5 below).
- Does not back up the DB first. **Run a Supabase backup snapshot before invoking** if there is any chance the data is recoverable-and-wanted.

---

## Pre-flight checklist

- [ ] Latest code is deployed to Render and the service is UP (`/api/health` returns `{"ok": true, "db": true}`).
- [ ] You have a recent Supabase backup snapshot (Database → Backups → "Take a backup now" if needed).
- [ ] You have a strong password (12+ chars) ready for the first CEO.
- [ ] You are physically at your laptop with the Render dashboard open.

---

## Step 1 — Set the four env vars in Render

In the Render dashboard for the `ufit-motion` service:

1. Open **Environment**.
2. Add (or update) these four env vars:
   - `SUPER_ADMIN_EMAIL` → e.g. `ceo@ufitonline.net`
   - `SUPER_ADMIN_PASSWORD` → strong, 12+ chars
   - `SUPER_ADMIN_FIRST` → e.g. `Miss`
   - `SUPER_ADMIN_LAST` → e.g. `A`
3. Click **Save Changes** — Render will redeploy automatically. Wait for green.

---

## Step 2 — Run the wipe via Render Shell or Job

**Option A — Render Shell (one-click from dashboard):**

1. Open the service → **Shell** tab.
2. Run:

   ```bash
   python -m scripts.production_wipe --yes-really
   ```

3. Watch stderr for `[wipe] complete. CEO seeded as <email>.`

**Option B — Render Job (declarative, audit-able):**

Add a one-shot Job to `render.yaml` (or create via dashboard):

```yaml
- type: cron
  name: production-wipe-oneshot
  env: python
  schedule: "0 0 31 2 *"   # never fires (Feb 31)
  buildCommand: pip install -r requirements.txt
  startCommand: python -m scripts.production_wipe --yes-really
```

Trigger it manually from the dashboard. Same env vars must be present.

---

## Step 3 — Verify the wipe succeeded

1. Visit `/login` on production.
2. Log in via the **Admin** portal with `SUPER_ADMIN_EMAIL` / `SUPER_ADMIN_PASSWORD`.
3. Confirm the dashboard loads with empty state cards (no schools, no coaches, no students).
4. Confirm `/api/admin/audit-log` shows zero entries (or just the one row generated by your login).

If login fails, check Render logs for `[wipe]` lines — most failures are a missing env var or a password that doesn't meet the 12-char minimum.

---

## Step 4 — Change the CEO password through the app

The env-var-supplied password is now sitting in three places:

- The Render env vars
- Your terminal scrollback / clipboard
- (If you Slacked it to anyone) message history

Reduce the surface area:

1. From the logged-in CEO account, open **Account → Change Password**.
2. Set a new password.

This invalidates the env-var password as the source of truth.

---

## Step 5 — Scrub the env vars from Render

In the Render dashboard → Environment:

1. Delete `SUPER_ADMIN_PASSWORD`.
2. Optionally delete the other three (`EMAIL`, `FIRST`, `LAST`) — they're not secrets but they're dead weight.
3. Save Changes. Render will redeploy.

After this redeploy, the wipe script will refuse to run again (missing env vars → exit code 3) until someone explicitly re-sets them.

---

## Step 6 — Document the run

Append a one-line entry to `docs/runbooks/operational-log.md` (create the file if missing):

```
2026-MM-DD — production wipe ran, CEO bootstrapped as <email>. By: <your name>
```

This builds the audit history outside the app's `audit_log` table (which the wipe just truncated).

## Failure modes & recovery

| Symptom | Likely cause | Fix |
|---|---|---|
| Exit code 2 | `APP_ENV` is not `"production"` on Render | Confirm Render env has `APP_ENV=production` |
| Exit code 3 | At least one `SUPER_ADMIN_*` env var unset | Re-add the missing var, redeploy, retry |
| Exit code 4 | Password < 12 chars | Use a longer password |
| Exit code 5 | Forgot `--yes-really` flag | Re-run the command with the flag |
| Exit code 1 | Database error (FK conflict, connection failure) | Check Render logs; likely Supabase pooler timeout — retry. If a TRUNCATE is partially-applied (it shouldn't be — we use a single transaction), restore from the snapshot taken in pre-flight. |
| Login works but no skills / domains visible | The wipe accidentally hit a preserved table | Restore from snapshot. File a bug — `PRESERVED_TABLES` in `scripts/production_wipe.py` should match what `init_db()` re-seeds. |

---

## Why it's atomic

The Postgres path issues a single `TRUNCATE … RESTART IDENTITY CASCADE` followed by an `INSERT INTO users`, both inside one transaction (psycopg3 autocommit=False, manual `db.commit()` at the end). If the INSERT fails, the TRUNCATE is rolled back and the database is unchanged. This is intentional — there is no acceptable window of "zero admins" on a production system.

The SQLite test path is best-effort (we run `PRAGMA foreign_keys = OFF` and DELETE per table) because SQLite doesn't have the same multi-table TRUNCATE primitive. Production never uses this path.
