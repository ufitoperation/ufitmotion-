System Reference

A guide to how Ufit Motion is built — the data model, how the four user roles interact, the lifecycle of every important action (session, incident, onboarding), and the operational shape of the platform. Aimed at administrators, future engineers, and anyone evaluating Ufit Motion technically.

1. Overview

Ufit Motion is a multi-organization PE program management platform. One installation serves multiple school districts, with strict data isolation between organizations. Inside an organization, four user roles consume and produce data:

RoleWhat they doWhat they see
Admin / CEOOnboard schools, hire coaches, monitor compliance, export reportsEverything across the organization
CoachRun sessions, log EODs, score assessments, file incidentsTheir assigned schools only
Principal / School StaffReview EODs, monitor program health at their schoolTheir one school only (read-mostly)
ParentView their child's progressTheir child's record only

The platform is opinionated: every screen exists for a reason. There are no dashboards "you can customize," no "themes," no "workflow builders." The product is a service-delivery tool, not a no-code platform.

2. Architecture

Ufit Motion is a single Flask web application backed by Postgres, hosted on Render, with a Supabase-managed database.

LayerTechnologyWhy
BackendFlask 3 (Python 3.12)Mature, simple, well-understood. Single-process gunicorn under Render.
DatabasePostgreSQL 15+ via SupabaseManaged Postgres with automatic backups, point-in-time recovery, and a transaction pooler for connection multiplexing.
FrontendVanilla JavaScript SPANo framework dependency. The entire frontend lives in one static/app.js file and one templates/index.html shell. Cold load is small (<200 KB).
HostingRender (web service + cron)Auto-deploy from main branch on push. One nightly cron recalculates coach performance scores.
EmailGmail SMTP (paid Render) or Resend HTTP API (free Render)Dual transport — code prefers Resend if configured, falls back to Gmail SMTP.
AuthFlask sessions (signed-cookie)HTTP-only, SameSite=Strict, secure flag in production. PBKDF2-SHA-256 password hashing.
CRM syncHubSpot integration (optional)New principals + parents are mirrored to HubSpot for outreach tracking.
Single source of truth. All operational data lives in one Supabase Postgres database. There are no microservices, no cross-database joins, no eventual-consistency edge cases. A query against the production DB returns the truth at the time of the query.

3. Data Model

32 tables in total, organized into four conceptual groups:

Identity (who's in the system)

  • organizations — the top-level tenant boundary. Every other row is scoped to one of these.
  • regions — geographic groupings inside an organization.
  • contracts — the legal agreement between Ufit and a school district.
  • users — the canonical identity row. Joined to staff_profiles or parents depending on role.
  • staff_profiles — Ufit-employee details (employee type, hire date, training level).
  • staff_assignments — which staff member works at which school (multi-school capable).
  • parents — parent records, linkable to up to two students each.

Programs & sessions (what Ufit delivers)

  • schools — physical school sites.
  • programs — the contracted service at a school (PE Support, Lunch Sports, After-School Sports, etc.).
  • sessions — a single coaching session: date, time, location, lead coach, planned vs actual activities.
  • session_staff — which staff worked which session (lead, assistant).
  • student_session_attendance — which students attended, with participation level.
  • students — the student roster, including local_student_identifier (e.g. LAUSD-2026-0001).
  • student_program_enrollment — which students are enrolled in which program.

Skills & assessments (what students can do)

  • skill_domains — the top-level skill categories (Physical/Psychomotor, Sport Fundamentals, SEL/Behavioral).
  • skills — individual skills (dribbling, chest pass, listening) tied to a domain.
  • benchmarks — the 1-5 rubric definitions per skill per grade band.
  • assessment_windows — time periods for batch assessment (Fall, Winter, Spring).
  • assessments — a single assessment event for one student.
  • assessment_scores — individual skill scores within an assessment (1-5 raw, 20-100 normalized).
  • behavior_observations — lightweight per-session SEL ratings (teamwork, effort, self-control).

Operations & reporting (what's happening)

  • eod_reports — end-of-day reports filed by coaches.
  • incident_reports — injuries, behavior issues, equipment failures, facility issues.
  • coach_observations — supervisor scoring of coaches on 6 dimensions.
  • coach_evaluations — formal periodic evaluations of coaches.
  • coach_performance_snapshots — nightly cron-computed rolling performance score per coach.
  • school_reports — generated PDF reports for principals.
  • principal_satisfaction_surveys — quarterly principal feedback.
  • student_skill_summary, student_domain_summary, student_overall_summary — calculated rollups for parent progress views.
  • notifications — in-app bell-icon feed.
  • audit_log — write-only log of every state-changing action.
  • app_settings, role_permissions — system configuration.

Defense in depth. Every API endpoint enforces the role check at the route handler level. Every database query that touches student records also includes an organization or school scope filter. Frontend route guards exist but are never the only line of defense.

5. Communication Flow

Here's how data actually moves between the four roles. Each subsection below traces a complete cycle.

5a. Session lifecycle (coach → admin → principal → parent)

A typical PE session, from start to surfacing in the parent's view
Coach→Logs in, picks today's school (if multi-assigned)
Coach→Starts session: planned activities, student group, time
Coach→Marks attendance for each student (present / absent / partial)
Coach→Scores assessments (1-5 + observation tag) for any students assessed today
Coach→Files end-of-day report (activities, engagement, behavior, success, challenges)
System→Recalculates student skill / domain / overall summaries (nightly cron + on-write triggers)
Admin→Sees the EOD on the admin dashboard; compliance % updates
Principal→Sees the latest EOD on her dashboard the next time she logs in
Parent→Sees updated baseline / current / growth in the child's progress view

5b. Incident lifecycle (coach → admin → school → family)

From "student turned ankle" to closed incident
Coach→Files incident: type (injury/behavior/equipment/facility), severity, description, action taken
Coach→Toggles "school notified" + "family notified" booleans
System→If severity = critical: writes a notification row for every admin / overseer in the organization
System→audit_log row written: action=incident_filed, table=incident_reports
Admin→Sees critical incident at the top of the admin dashboard, opens the detail view
Admin→Reviews; may add resolution notes; closes the incident
Principal→Sees the incident in her own school's incident feed for FERPA / duty-of-care compliance

5c. Coach onboarding (three converging paths)

Three ways a coach gets into the system; all converge on the same email → set-password flow
Admin→Path 1: Add Coach modal, single coach entry
Admin→Path 2: Bulk Invite modal, paste up to 200 rows
Coach→Path 3: Coach Portal → "Sign up with school code" — uses code admin gave the school
System→Creates pending user (active_status=FALSE, no password_hash) + staff_profile + staff_assignment
System→Generates 24-hour invite token; sends "Set My Password" email via Gmail / Resend
Coach→Clicks email link, sets password, account becomes active_status=TRUE
Coach→Logs in, lands at picker (if multi-school) or dashboard

5d. Parent self-registration (parent → student verification → access)

Parent gets to their child's progress without staff help
Parent→Visits Parent Portal → "Create Account"
Parent→Step 1: enters child's first/last name + local_student_identifier (e.g. LAUSD-2026-0001)
System→Verifies the student exists with that exact identifier and name
Parent→Step 2: enters parent name / email / phone / relationship / password
System→Creates user + parent row; links to student.parent_primary_id (or _secondary if primary already filled)
System→Auto-login + writes notification to admins; HubSpot sync fires in background
Parent→Lands on child's progress view: baseline / current / growth / behavior readiness
Edge case: both parent slots full. If a student already has parent_primary_id and parent_secondary_id set, a third parent self-registering creates the parent record but leaves it unlinked, with a note. An admin notification fires for manual resolution.

6. Authentication & Security

  • Passwords — hashed with PBKDF2-SHA-256 (1,000,000 iterations). Plain passwords never touch the database.
  • Sessions — Flask signed-cookie sessions; HTTPOnly, SameSite=Strict, Secure in production. 8-hour idle timeout.
  • CSRF — every POST/PATCH/DELETE against /api/* requires the X-Requested-With: XMLHttpRequest header (set by the SPA's fetch wrapper). Plain HTML form posts from a phisher's domain are rejected.
  • Rate limiting — Flask-Limiter on sensitive endpoints (login: 10/min, forgot-password: 5/min, coach-register: 5/hour, bulk-invite: 10/min).
  • Constant-time login — even on "user not found", the server runs a dummy password hash check to keep response time uniform (prevents timing-based email enumeration).
  • Audit log — every state-changing action writes a row to audit_log with the actor's user_id, IP, action verb, table, record_id, old_values, new_values.
  • Session fixation — login clears the old session and rotates the session ID. Logout fully clears the session.
  • Security headers — every response includes X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Content-Security-Policy, Strict-Transport-Security in production.

Transport: Resend HTTP API (preferred, works on free Render) or Gmail SMTP (fallback, works on paid Render or local dev). Auto-selected based on which env vars are set.

Notifications (in-app)

Bell-icon feed in the top-nav. Currently fires on:

  • Critical incident filed (recipients: all admins / CEOs / overseers in the org)
  • Parent self-registers (recipients: same)
  • Coach self-registers via school code (recipients: same)
  • Pending invite about to expire (planned — needs a scheduled job)

8. FERPA & Compliance

Student records on Ufit Motion are "education records" under FERPA (20 U.S.C. § 1232g). Ufit operates as a "school official" with legitimate educational interest under each district's data-processing agreement.

  • Org boundary — every student-touching query includes an organization or school filter. A coach in District A literally cannot query District B's students; the SQL won't return them.
  • Parent rights — parents can inspect, correct, or request deletion of their child's records. Requests are routed through the school's FERPA officer.
  • Retention — student records are retained for the duration of the school's contract + 3 years for audit purposes. Earlier deletion requested via the school.
  • Re-disclosure — Ufit does not redisclose student data to any party outside the school's data-processing agreement, with exceptions only as required by law.
  • Audit trail — every read or write of student data is auditable; the audit_log table captures actor, action, record, timestamp, IP.
